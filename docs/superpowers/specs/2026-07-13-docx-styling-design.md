# Instructor Guide DOCX: Full Front-Matter Styling

## Problem

The generated `.docx` today (`buildInstructorGuideDocx(slides, title)`) is just a title
page followed by per-slide sections. The user has a reference instructor-guide document
(`output/instructor-guide-6ad5ad14-751c-412d-a63a-ed197b2fbc88 (1).docx`) with a much
richer front matter: Workshop Title, Duration, Learning Objectives, Trainer Guidelines
(Do's/Don'ts table), Material Required, Training Aids, and Training videos/links —
*before* the per-slide "Session Guide" content. The export needs to be restyled to match
that structure.

## Goals

- Workshop Title is derived from the deck's own content (title/welcome slide), not just
  the uploaded filename — falling back to the filename if no title is stated.
- Duration is filled in only if a slide explicitly states one (verbatim, no inference);
  otherwise the label is rendered with blank space for the user to fill in by hand.
- Learning Objectives (3–5) are generated for the whole deck, each starting with an
  imperative/base-form verb (e.g. "Understand", "Apply", "Identify") — never a gerund
  ("Understanding...", "Learning...").
- Trainer Guidelines (Do's/Don'ts table), Material Required, Training Aids, and Training
  videos/links are **static** content, identical across every generated guide, sourced
  from the reference document — never generated by Gemini.
- Document section order: Workshop Title → Duration → Learning Objectives → Trainer
  Guidelines → Material Required → Training Aids → Training videos → "Session Guide" →
  existing per-slide content (unchanged).

## Non-Goals

- No UI for editing the static content (Trainer Guidelines etc.) — it's a hardcoded
  module; changing it requires a code change and redeploy, same as any other constant
  in this codebase.
- No retry/regeneration UI for the new deck-level fields — if `analyzeDeck` fails, the
  job still completes; the fields are simply left blank in the export (matches the
  existing per-slide failure-isolation philosophy).
- No validation/enforcement in code of the "non-gerund verb" rule — enforced via prompt
  instruction only, per user decision.
- No change to per-slide analysis/generation (`analyzeSlide`, `generateGuide`) or their
  prompts.

## Architecture

### Database

**`src/lib/schema.sql`** — add three nullable columns to the `Job` table definition
(new databases created via `CREATE TABLE IF NOT EXISTS` get them for free):

```sql
"workshopTitle" TEXT,
"duration" TEXT,
"learningObjectives" TEXT,  -- JSON-encoded string[]
```

Since `CREATE TABLE IF NOT EXISTS` is a no-op against an already-existing `Job` table
(a known, previously-documented limitation from the Postgres migration), `schema.sql`
must also append explicit statements so existing databases (dev + test) pick up the
new columns the next time `scripts/migrate.mjs` runs:

```sql
ALTER TABLE "Job" ADD COLUMN IF NOT EXISTS "workshopTitle" TEXT;
ALTER TABLE "Job" ADD COLUMN IF NOT EXISTS "duration" TEXT;
ALTER TABLE "Job" ADD COLUMN IF NOT EXISTS "learningObjectives" TEXT;
```

**`src/lib/db.ts`** — extend `JobRow` with:
```ts
workshopTitle: string | null;
duration: string | null;
learningObjectives: string | null; // JSON string[]
```
`db.job.create` gains no new required args (all three start `null` until
`analyzeDeck` runs); `db.job.update` already accepts arbitrary partial data via
`buildSetClause`, so no signature change needed there.

### New Gemini call

**`src/lib/gemini.ts`** — add `analyzeDeck`:

```ts
export async function analyzeDeck(slideTexts: string[]): Promise<DeckAnalysis>;
```

- Text-only input: the concatenated OCR text of every slide (no images — title/duration/
  objectives are drawn from what the deck *says*, not its visual layout).
- New prompt (`DECK_ANALYZER_PROMPT`) instructing Gemini to:
  1. Find the workshop/session title if a slide explicitly states one; else `null`.
  2. Find an explicit duration/schedule statement (e.g. "2 hours", "9:30 AM – 5:00 PM",
     "Day 1 & Day 2") verbatim; else `null`. Explicit instruction: never infer or
     estimate a duration that isn't stated.
  3. Generate 3–5 learning objectives for the whole deck. Each must start with an
     imperative/base-form verb (Understand, Apply, Identify, Explain, Analyze,
     Evaluate, Describe, Create, Demonstrate, ...). Never start with a gerund/"-ing"
     form.
- JSON-schema-constrained response (`SchemaType.OBJECT` with `workshopTitle`,
  `duration`, `learningObjectives` — mirroring the existing `analyzerResponseSchema`
  pattern in the same file).

**`src/lib/schemas.ts`** — add `deckAnalysisSchema` (zod), validating the parsed JSON:
```ts
z.object({
  workshopTitle: z.string().nullable(),
  duration: z.string().nullable(),
  learningObjectives: z.array(z.string()).min(3).max(5),
})
```

**`src/types/guide.ts`** — add:
```ts
export interface DeckAnalysis {
  workshopTitle: string | null;
  duration: string | null;
  learningObjectives: string[];
}
```

### Worker integration

**`src/lib/worker.ts`** — in `processJob`, after `extractSlideTexts` resolves (texts for
all slides are available) and before/alongside the per-slide `Promise.all`:

```ts
try {
  const deckAnalysis = await analyzeDeck(texts);
  await db.job.update({
    where: { id: jobId },
    data: {
      workshopTitle: deckAnalysis.workshopTitle,
      duration: deckAnalysis.duration,
      learningObjectives: JSON.stringify(deckAnalysis.learningObjectives),
    },
  });
} catch (err) {
  // Deck-level analysis failure must not fail the whole job — the export falls back
  // to filename-as-title and blank duration/objectives. Per-slide processing continues
  // regardless.
}
```

This runs once per job, independent of the per-slide `pLimit(3)` pool (it's a single
call, not one-per-slide).

### `docx-export.ts` restructuring

Signature changes from `buildInstructorGuideDocx(slides, title)` to
`buildInstructorGuideDocx(job, slides)` (needs `job.workshopTitle`, `job.duration`,
`job.learningObjectives`, plus the existing filename-based fallback logic).

New document body order:

1. **Title page**: `job.workshopTitle ?? stripPptxExtension(job.filename)` as
   `HeadingLevel.TITLE`.
2. **Duration**: a paragraph `"Duration: " + (job.duration ?? "")` — if `null`, the
   label is rendered with trailing blank space (no placeholder text like "TBD"), so the
   user can fill it in by hand in Word, matching the reference doc's blank-cell pattern.
3. **Learning Objectives**: heading + one bullet `Paragraph` per string in
   `JSON.parse(job.learningObjectives ?? "[]")`.
4. **Trainer Guidelines**: heading, then a 2-column `docx.Table` built from
   `TRAINER_GUIDELINES_DOS`/`TRAINER_GUIDELINES_DONTS` (header row "Do's" / "Don'ts",
   then one row per index up to the longer list's length, empty cell where the shorter
   list runs out — matching the reference doc's uneven table).
5. **Material Required for the Workshop**: heading + bullet list from
   `MATERIAL_REQUIRED_ITEMS`.
6. **Training Aids for the Workshop**: heading + bullet list from
   `TRAINING_AIDS_ITEMS`.
7. **Training videos and important links**: heading + bullet list from
   `TRAINING_VIDEO_ITEMS`.
8. **"Session Guide"** heading (`HeadingLevel.HEADING_1` or similar, visually separating
   front matter from per-slide content).
9. Existing per-slide paragraphs (`slideToParagraphs`) — unchanged.

**`src/app/api/jobs/[id]/export/route.ts`** — currently fetches only `slides` and calls
`buildInstructorGuideDocx(slides, stripPptxExtension(job.filename))`. Needs to also
fetch the `job` row (already does, for `job.filename`) and pass the whole `job` object
through instead of a pre-extracted title string.

## Error Handling

- `analyzeDeck` throws (malformed Gemini JSON, network error, etc.) → caught in
  `worker.ts`, job continues, fields stay `NULL` in the DB. Export then falls back to
  filename-as-title, blank duration, and an empty Learning Objectives list (heading
  still renders, with no bullets under it).
- `job.learningObjectives` is `NULL`/unparseable at export time → treated as `[]`,
  matching the existing `slide.sections ? JSON.parse(...) : []` defensive pattern
  already used in `docx-export.ts` and the guide page.
- Existing per-slide error handling (`slide.status === "failed"`) is unchanged.
