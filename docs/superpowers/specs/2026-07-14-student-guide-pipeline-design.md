# Student Guide (SG) Pipeline

## Problem

Today the app only generates an Instructor Guide (IG): per-slide trainer-facing
content (trainer pointer, mental model, best practices, pitfalls, FAQ, etc.)
plus a workshop front matter with trainer logistics (guidelines table, material
required, training aids, videos).

The user wants a second, learner-facing guide type — a Student Guide (SG) —
generated from the same uploaded deck, with a different per-slide structure and
a slimmer front matter (workshop title/duration/learning objectives, but no
trainer-logistics material).

Per a reference few-shot document (`fewshot examples of SG.docx`), each SG
slide entry has:

1. **Core Explanation** — either a *Concept Explanation* (slide is primarily
   textual) or a *Visual Walkthrough* (slide is primarily visual: diagram,
   architecture, process, workflow, chart, graph, image, table).
2. **Remember This** — 2-3 crisp bullets.
3. **Mental Model** — one real-life analogy.
4. **Self-Probing Questions** — 2-3 questions for the learner to test their own
   understanding (no answers, unlike IG's FAQ).

Per the user's clarification: every slide gets a Core Explanation, but
non-teaching slides (Welcome, Agenda, Section Divider, Thank You) get *only*
the Core Explanation — Remember This, Mental Model, and Self-Probing Questions
are skipped for those, mirroring how IG already trims sections for the same
slide types. Non-teaching slides are always treated as textual (Concept
Explanation), skipping the visual-detection step for them.

## Goals

- User can select Instructor Guide, Student Guide, or both at upload time, for
  a single deck/upload.
- SG reuses the existing PPTX-to-image conversion, OCR extraction, and
  deck-level analysis (`analyzeDeck`: workshop title, duration, learning
  objectives) — no duplicate work.
- SG reuses IG's existing per-slide `analyzeSlide()` call for `slideIntent`
  when both guides are requested, avoiding a duplicate classification call.
- A new lightweight classifier determines `contentMode` (`TEXTUAL` | `VISUAL`)
  for teaching slides only.
- A new SG generator produces the 4 SG sections (or just Core Explanation, for
  non-teaching slides), grounded in the slide — never generic filler,
  consistent with IG's existing prompt discipline.
- SG-specific DOCX export: same NIIT header/footer as IG, front matter with
  title/duration/learning objectives only (no trainer guidelines/material/aids/
  videos), then per-slide Core Explanation + Remember This + Mental Model +
  Self-Probing Questions.
- Web guide page shows export link(s) only for the guide type(s) actually
  requested for that job.

## Non-Goals

- No retroactive SG generation for jobs created before this change — existing
  `Job` rows get `guideTypes = '["ig"]'` via column default, so their behavior
  is unchanged.
- No merging of IG and SG into a single combined export document.
- No change to IG's existing prompts, schema, or export beyond adding the
  `guideTypes`-gated skip in the worker.
- No backfill/migration tooling for `Slide.contentMode` / `Slide.sgSections` on
  already-processed slides.

## Architecture

### Types (`src/types/guide.ts`)

```ts
export const CONTENT_MODES = ["TEXTUAL", "VISUAL"] as const;
export type ContentMode = (typeof CONTENT_MODES)[number];

export const GUIDE_TYPES = ["ig", "sg"] as const;
export type GuideType = (typeof GUIDE_TYPES)[number];

export const NON_TEACHING_INTENTS: SlideIntent[] = [
  "WELCOME",
  "AGENDA",
  "SECTION_DIVIDER",
  "THANK_YOU",
];

export const SG_SECTION_KEYS = [
  "coreExplanation",
  "rememberThis",
  "mentalModel",
  "selfProbingQuestions",
] as const;
export type SgSectionKey = (typeof SG_SECTION_KEYS)[number];

export interface StudentGuide {
  sections: GuideSection[]; // same GuideSection shape IG already uses
}

export const SG_SECTION_TITLES: Record<string, string> = {
  coreExplanation: "Core Explanation",
  rememberThis: "Remember This",
  mentalModel: "Mental Model",
  selfProbingQuestions: "Self-Probing Questions",
};
```

`coreExplanation` sections carry their concrete flavor ("Concept Explanation"
or "Visual Walkthrough") in `GuideSection.title`, generated by the model
per-slide; `SG_SECTION_TITLES.coreExplanation` is the display fallback used by
`sectionDisplayTitle()` only if the model's title is empty. `rememberThis`
uses `keyPoints: string[]` (2-3 bullets). `mentalModel` uses `content: string`.
`selfProbingQuestions` uses `keyPoints: string[]` (2-3 plain question
strings — no `items`/answers, since these are for the learner to self-test,
not an answered FAQ).

### Zod validation (`src/lib/schemas.ts`)

```ts
export const contentModeSchema = z.object({
  contentMode: z.enum(CONTENT_MODES),
});

export const studentGuideSchema = z.object({
  sections: z.array(guideSectionSchema),
});
```

Reuses the existing `guideSectionSchema` — no new section shape needed.

### Gemini (`src/lib/gemini.ts`)

**`classifyContentMode(imageBase64, extractedText)`** — new small analyzer
call, invoked only when the slide's `slideIntent` (from the shared
`analyzeSlide()` result) is not in `NON_TEACHING_INTENTS`:

```
CONTENT_MODE_PROMPT = `You are an expert Instructional Designer.

Determine whether this slide is primarily TEXTUAL (mostly prose, bullet
points, definitions) or primarily VISUAL (diagram, architecture, process,
workflow, chart, graph, image, table -- where understanding the slide
requires interpreting the visual, not just reading text).

Return ONLY valid JSON. No explanation. No markdown.`
```

Response schema: `{ contentMode: "TEXTUAL" | "VISUAL" }`.

**`generateStudentGuide(imageBase64, extractedText, slideIntent, contentMode)`**
— new generator call:

- If `slideIntent` is in `NON_TEACHING_INTENTS` (`contentMode` is `null` in
  this case, since the classifier is skipped): generate only
  `coreExplanation`, as a short Concept Explanation.
- Otherwise: generate all 4 sections. `coreExplanation`'s flavor follows
  `contentMode` (`TEXTUAL` -> Concept Explanation, `VISUAL` -> Visual
  Walkthrough), matching the few-shot doc's style (explain what's shown,
  ground every claim in the slide's actual content/labels/numbers).

```
SG_GENERATOR_PROMPT = `You are an expert Instructional Designer writing a
Student Guide entry for a learner reading independently (not a trainer script).

Section Rules:

coreExplanation: If contentMode is TEXTUAL, write a Concept Explanation: a
clear paragraph explaining the concept the slide teaches, in the learner's own
study voice. If contentMode is VISUAL, write a Visual Walkthrough: explain
what the diagram/chart/process/table shows and what it means, walking through
its parts in order. Ground every sentence in the slide -- never invent
information. If this is a non-teaching slide (no contentMode provided), write
one short paragraph explaining what this slide is / why it's here, nothing
more.

rememberThis: Exactly 2-3 crisp bullets capturing the single most important
takeaways from this slide. Each bullet is a short, standalone, memorable
statement -- not a summary sentence.

mentalModel: One memorable real-life analogy that makes the concept concrete.
Only if a natural analogy exists -- do not force one.

selfProbingQuestions: 2-3 questions a learner should ask themselves to check
their own understanding of this slide. Questions only, no answers.

General Rules: Never invent information. Never generate generic filler.
Generate ONLY the requested sections. Whenever a section rule asks for
multiple bullets/questions, each one MUST be its own array item -- never
merge multiple points into one string. NEVER use an em dash (—) anywhere
in any generated text; use a comma, period, or parentheses instead. Return
ONLY valid JSON.`
```

Response schema mirrors `generatorResponseSchema` (reused as-is — same
`{ sections: [...] }` shape). Result is validated with `studentGuideSchema`
and sanitized with the existing `sanitizeSection`/`stripEmDash` helpers.

### Worker (`src/lib/worker.ts`)

`processJob`: parses `job.guideTypes` (JSON array, default `["ig"]` for
pre-existing rows). Passes it through to `processSlide`.

`processSlide(slideId, jobId, guideTypes)`:

1. Always calls `analyzeSlide()` (needed for `slideTitle` display and, when
   `"ig"` is requested, `recommendedSections`; when only `"sg"` is requested,
   only `slideIntent`/`slideTitle` are used from it).
2. If `"ig" in guideTypes`: calls `generateGuide(...)` as today, writes
   `slideIntent`, `recommendedSections`, `sections`.
3. If `"sg" in guideTypes`: if `analysis.slideIntent` is a teaching intent,
   calls `classifyContentMode(...)` first, then `generateStudentGuide(...)`
   with that `contentMode`; if non-teaching, calls `generateStudentGuide(...)`
   with `contentMode = null` directly (skipping the classifier call). Writes
   `contentMode` and `sgSections`.
4. `slideTitle` is written once regardless of which guide type(s) ran.

Both generation branches run in parallel via `Promise.all` when both guide
types are requested. A failure in either branch still marks the slide
`"failed"` with the error message (same all-or-nothing per-slide failure
semantics as today — not split per guide type, to keep retry logic simple).

### DB schema (`src/lib/schema.sql`, `src/lib/db.ts`)

```sql
ALTER TABLE "Job" ADD COLUMN IF NOT EXISTS "guideTypes" TEXT NOT NULL DEFAULT '["ig"]';
ALTER TABLE "Slide" ADD COLUMN IF NOT EXISTS "contentMode" TEXT;
ALTER TABLE "Slide" ADD COLUMN IF NOT EXISTS "sgSections" TEXT;
```

`JobRow` gains `guideTypes: string`. `SlideRow` gains `contentMode: string |
null` and `sgSections: string | null`. `db.job.create()` accepts an optional
`guideTypes` field (defaults to `'["ig"]'` if omitted, matching the column
default). `db.slide.create()`/`update()` need no signature change beyond the
existing generic `data: Record<string, unknown>` pass-through already used by
`update()`; `create()` gains optional `contentMode`/`sgSections` params
alongside the existing IG ones.

### Upload (`src/components/UploadForm.tsx`, `src/app/api/upload/route.ts`)

`UploadForm`: two checkboxes, "Instructor Guide" (checked by default) and
"Student Guide". Submission blocked client-side if neither is checked. Posts
`guideTypes` (JSON-stringified array of the checked values, in `["ig","sg"]`
order) as an additional form field.

`POST /api/upload`: reads `guideTypes` from the form data, validates it's a
non-empty subset of `["ig","sg"]` (falls back to `["ig"]` if missing/invalid,
preserving current single-guide behavior for any existing client), passes it
to `db.job.create()`.

### Export (`src/lib/docx-export.ts`, new route)

New `buildStudentGuideDocx(job: JobRow, slides: SlideRow[]): Promise<Buffer>`,
structured like `buildInstructorGuideDocx` but with its own front matter and
section renderer:

```ts
function sgFrontMatter(job: JobRow): (Paragraph | Table)[] {
  const title = job.workshopTitle ?? stripPptxExtension(job.filename);
  const learningObjectives: string[] = job.learningObjectives ? JSON.parse(job.learningObjectives) : [];
  return [
    new Paragraph({ heading: HeadingLevel.TITLE, children: [new TextRun(title)] }),
    new Paragraph({ children: [new TextRun(`Duration: ${job.duration ?? ""}`)] }),
    heading("Learning Objectives"),
    ...bulletParagraphs(learningObjectives),
    heading("Student Guide"),
  ];
}
```

Per-slide rendering reuses the same `Slide N: <title>` heading and slide-image
embedding as IG's `slideToParagraphs`, but iterates `slide.sgSections` (parsed
JSON) through a new `sgSectionToParagraphs(section: GuideSection):
Paragraph[]` that mirrors `sectionToParagraphs` (heading via
`sgSectionDisplayTitle`, content via `parseMarkdownLite`, `keyPoints` via
`bulletParagraphs`) — no `items`/Q&A rendering needed since SG sections never
use `items`.

`sgSectionDisplayTitle(section)`: same pattern as `sectionDisplayTitle`, but
looks up `SG_SECTION_TITLES` first; for `coreExplanation` it falls through to
the model's own `title` (which already says "Concept Explanation" or "Visual
Walkthrough").

New route `src/app/api/jobs/[id]/export/sg/route.ts`, structurally identical
to the existing `.../export/route.ts` but calling `buildStudentGuideDocx` and
using filename `student-guide-<jobId>.docx`. The existing `.../export/route.ts`
is left unchanged (IG export keeps its current URL).

### Web guide page (`src/app/guide/[jobId]/page.tsx`, `SlideCard`)

Parses `job.guideTypes`. Renders "Export Instructor Guide (.docx)" link only
if `"ig"` present, "Export Student Guide (.docx)" link only if `"sg"` present.

`SlideCard` gains an optional `sgSections` prop. When both `sections` (IG) and
`sgSections` (SG) are present for a slide, render them as two labeled blocks
("Instructor Guide" / "Student Guide") stacked in the existing card, reusing
the current per-section rendering logic for each list.

## Error Handling

- `job.guideTypes` missing/unparseable on an old row → treated as `["ig"]`
  (matches the column default; keeps pre-existing jobs exporting IG only).
- `classifyContentMode` failure → the whole slide's SG generation fails for
  that slide (caught by `processSlide`'s existing try/catch, marks slide
  `"failed"`), same as any other per-slide generation failure today. No silent
  fallback content mode, to avoid silently mislabeling a Visual Walkthrough
  slide as a Concept Explanation or vice versa.
- SG export requested for a job whose `guideTypes` didn't include `"sg"` →
  `sgSections` will be `null` for every slide; the export still succeeds but
  produces a document with only front matter and empty per-slide sections
  (acceptable: the UI never surfaces this link when `"sg"` wasn't requested,
  so this only matters for someone hitting the URL directly).
